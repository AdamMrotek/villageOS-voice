2 · One state machine drives every pixel

status + isSpeaking + a tool-pending flag collapse into a single VoiceState. The orb, the labels, and the timeline all render from it.

idle tap to talk
→
connecting minting token
→
listening hearing you
→
thinking tool in flight
→
speaking agent talks
↺
error

The non-obvious bit: the SDK exposes no “thinking” state. It’s inferred — set true while get_schedule is in flight, cleared the instant the agent starts speaking (with a timeout fallback). That inferred state is what makes retrieval feel like the agent is doing something rather than freezing.

3 · The voice→UI handshake: one fetch, two payloads

The same /api/schedule result is split. The UI gets everything it needs to render; the model gets only what it will speak — because the tool result is re-sent to the LLM on every later turn, so unspoken fields are recurring token waste.

4 · An anti-corruption mapper around VillageOS’s schema

The source snapshot mirrors the real VillageOS API payload (snake_case). A mapper converts it to the voice app’s own stable contract, so when this swaps to a live query, only the mapper has to change.

VillageOS shape (snake_case)
VillageEvent
  • event_type, start_time / end_time
  • is_all_day, action_items[]
  • cost_estimate_gbp, urgent, done
  • seeded with relative dayOffset, resolved to real dates at request time

toScheduleEvent()
Voice app contract (camelCase)
ScheduleEvent
  • category, time / endTime
